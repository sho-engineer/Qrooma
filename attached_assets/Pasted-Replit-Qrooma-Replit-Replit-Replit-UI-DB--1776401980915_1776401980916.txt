Replitへ

Qrooma を Replitベースで進めたい です。
ただし、全部をReplit内で完結させるのではなく、Replitをフロント/UIと公開基盤の中心として使い、認証・DB・非同期実行は既定方針どおり Supabase + Trigger.dev を前提に進めてください。

プロジェクト概要

Qrooma は BYOK（Bring Your Own Key）型の非同期 AI チームルームアプリです。
ユーザーが質問を投げると、複数のAIが異なる視点で議論し、最後に結論を返します。

アーキテクチャ方針

Replit で担当すること
	•	フロントエンド実装
	•	UI/UX改善
	•	ランディングページ
	•	ルーティング
	•	ドッグフーディング用の公開先
	•	本実装に繋がる画面と状態設計

Replit で無理に持たないこと
	•	Auth を Replit 独自実装に寄せること
	•	DB を Replit専用構成に置き換えること
	•	非同期ジョブ基盤を Replit上で自前再構築すること

既定の本実装先
	•	Auth / DB / RLS: Supabase
	•	非同期議論実行: Trigger.dev
	•	AI呼び出し: OpenAI / Anthropic / Google
	•	APIキー保存: サーバーサイド暗号化前提

今回やってほしいこと

1. Replit上でQroomaのUI/導線を完成度高く保つ

以下の画面を Replit 上でしっかり動く状態にしてください。
	•	/ ランディング
	•	/login
	•	/signup
	•	/rooms
	•	/rooms/:id
	•	/settings

2. 今のUIを本実装前提で整理する

ダミーでもよいですが、Supabase / Trigger.dev に接続しやすい構造 にしてください。

具体的には：
	•	ローカル状態やダミーデータに依存しすぎない
	•	将来 API に差し替えやすいように責務を整理
	•	画面コンポーネントとデータ取得責務を分ける
	•	できればモックAPIレイヤー / service層を作る

3. Qrooma の体験を明確にする

Qroomaは「3モデルを並べる場」ではなく、複数の視点で議論して結論に至る場です。

そのためUI上の主表示は以下の形式に寄せてください。
	•	提案（GPT）
	•	検証（Claude）
	•	実行（Gemini）
	•	結論

英語UIでは
	•	Proposal (GPT)
	•	Review (Claude)
	•	Execution (Gemini)
	•	Conclusion

4. BYOK前提を分かりやすくする
	•	APIキー取得導線
	•	非ITの人向け説明
	•	ステップバイステップの設定案内
	•	APIキー未設定では議論が動かないこと
を UI に自然に組み込んでください。

5. 2 / 3 エージェント切替
	•	参加エージェント数 2 or 3
	•	default は 3
	•	2 の場合は Side A / B のみ
	•	3 の場合は Side A / B / C

6. 同一モデル重複禁止
	•	同じ provider + model は複数の枠で選べないようにする
	•	可能なら選択時点で防ぐ
	•	難しければ保存時エラー

7. ランディングとアプリ全体の見た目改善

方向性は以下です。
	•	Appleっぽい静かさ
	•	ただし白に黒を置いただけの平坦さは避ける
	•	軽いグラデーション
	•	typography hierarchy
	•	surface layering
	•	軽い motion
	•	AIっぽい派手さは抑える
	•	上質な業務ツール / 思考支援プロダクトに見せる

⸻

重要な制約

MVPでやらないこと
	•	画像アップロード
	•	ファイルアップロード
	•	添付プレビュー
	•	OCR
	•	共有ルーム
	•	課金
	•	高度な権限管理
	•	Replit独自DBへの全面置き換え

重要
	•	Replitベースで進めるが、Supabase / Trigger.dev 前提を崩さない
	•	今は「使える見た目」と「接続しやすい構造」の両立を優先する
	•	本実装の邪魔になる独自実装は避ける
