Qrooma を以下の内容で修正してください。
今回は、議論品質の細かい改善より先に、UIの完成度と意思決定フローの完成を優先してください。

現状認識

Qrooma は以前より改善しているが、まだ以下の問題が残っています。

残っている問題

1. 最終結論カード内の 次アクションの下部が見切れる
2. 2ラウンド終了後に「議論を続ける / ここで終える」ボタンが見当たらない
3. 以前からの不具合・改善要件も未完了または再確認が必要
    * 送信直後に結論セクションへ飛ぶ挙動
    * 日本語UIでの英語混在
    * Run 2以降の結論更新不整合
    * Casual / Soft なのにまだ少し硬い
    * 複合質問の要件分解不足
4. 議論が途中で一論点に寄りすぎて、元の質問全体をやや見失うことがある

⸻

目的

Qrooma を以下の状態にしてください。

* 最終結論カードが最後まで読める
* 2ラウンド終了後に、ユーザーが
    * ここで終える
    * 議論を続ける
        を明確に選べる
* 続けた場合は暫定結論が更新される
* 終えた場合はその時点の結論を最終版に確定する
* 日本語UIでは英語が混ざらない
* 複合質問を取りこぼさず拾う
* Casual / Soft では、やわらかいが判断力のある文体になる

⸻

1. 次アクションの下部が見切れる問題を修正

現象

* 最終結論カードの 次アクション セクション下部が切れている
* 最後まで読めない
* 固定フッターや下部UIと重なっている可能性がある

対応

以下を確認・修正してください。

* 結論カード親要素の overflow
* ページ全体スクロールと内部スクロールの競合
* 下部固定UI（入力欄 / プラン切替バー）分の bottom padding
* iPhone safe area を考慮した余白
* アコーディオン開閉時の高さ再計算

期待挙動

* 最終結論カードは最後まで読める
* 次アクション セクション全体が見える
* 下部UIに隠れない

推奨

* 画面下の固定入力欄・プランバーの高さを計算して、main scroll container に十分な padding-bottom を入れる
* 結論カード開閉時に scrollHeight を再評価する

⸻

2. 「議論を続ける / ここで終える」導線を実装

現状

設計上は必要だが、スクショ上ではボタンが見えない。
つまり、UIとして成立していない。

必須仕様

2ラウンド終了後は、自動で最終確定しない。
まず 暫定結論 を出し、その下に以下のボタンを表示する。

* ここで終える
* 議論を続ける

ボタンの意味

ここで終える

* 現在の暫定結論を 最終結論 に昇格する
* runStatus を completed にする
* 結論ラベルを 最終結論 にする

議論を続ける

* 現在の残論点を引き継いで追加ラウンドを行う
* 暫定結論を更新する
* 再度同じ2ボタンを出す

配置

以下のどちらか、または両方で実装してください。

A. 結論カード直下

* 最も自然
* 文脈的に分かりやすい

B. 入力欄上の固定アクションバー

* 見逃しにくい
* モバイルで押しやすい

期待挙動

* 2ラウンド後に、ユーザーが次のアクションを選べる
* ボタンが hidden / fold 下 / 見切れ状態にならない

⸻

3. 継続議論時の結論更新

問題

以前、Run 2 以降で結論が更新されない不具合があった。
この仕様では、継続議論のたびに暫定結論が更新される必要がある。

必須ルール

* 各 Run / 各 continuation ごとに conclusion を再生成する
* latest conclusion は常に最新 run のものを指す
* 前の結論は history に残す
* UI上の回数表示も更新する

データ構造

最低限以下を保持

* roomId
* runId
* conclusionId
* kind: provisional / final
* isLatest
* createdAt
* content
* previousConclusionId

期待挙動

* 続けるたびに暫定結論が更新される
* 終えると、その時点の最新版が final になる
* 結論履歴と最新表示がズレない

⸻

4. 送信直後に結論へ飛ぶ挙動を直す

問題

送信直後に 結論 生成中 のプレースホルダーへスクロールしてしまうことがあった

修正方針

* 送信直後は結論へ飛ばない
* 自動スクロール対象は「今生成中のAIメッセージ」
* 結論アコーディオンは自動展開しない
* 結論完成後も、自動で結論へ飛ばさない
* 必要なら「結論を更新しました」バッジだけ表示

⸻

5. 日本語UIで英語が混ざらないようにする

問題

以前、Gemini / critique / rebuttal が英語で出ていた

対応

* 日本語UIでは全表示テキストを日本語固定
* 内部テンプレや中間データが英語でも、表示前に必ず日本語化
* language guard を最終表示レイヤーに置く
* Agent A のような英語ラベルも 提案, 検証, 実行 など日本語化する

⸻

6. 複合質問の要件分解を維持・改善

問題

一度は改善したが、議論途中で一論点に寄りすぎることがある

対応

議論前に query decomposition を行い、内部で最低限以下を保持してください。

* エリア
* グルメ候補
* 観光候補
* 季節要件
* 移動条件
* 家族旅行条件
* 追加の固有要件（ホテル拠点など）

重要

議論中に一つの論点を深掘りしても、
元の質問全体の要件を捨てないこと

推奨

internal state に activeIssues だけでなく fullRequirementChecklist を持ち、
round summary と conclusion 生成時に参照する

⸻

7. Casual / Soft の文体をさらに自然にする

現状

前よりマシだが、まだ少し固い

目標

* やわらかい
* 自然
* 読みやすい
* でも判断する

NG寄り表現

* 「〜に依存している」
* 「〜の魅力は失われる」
* 「〜の満足度に影響を与える可能性があります」
* 「〜が懸念されます」

目指す表現

* 「この案の弱いところは、〜です」
* 「その条件だと、少し使いづらくなりそうです」
* 「ここはまだ詰めきれていません」
* 「今のところはA寄りですが、ここが決まれば変わる余地もあります」

重要

文体だけ柔らかくし、中身の判断力は落とさない

⸻

8. 結論の粒度を改善

現状

最終結論は出るが、まだ実行粒度が少し荒い

目標

人間の最終意思決定を助ける粒度まで落とす

改善ルール

最終結論では、できる限り以下を意識する

* 具体候補
* 優先順位
* 棄却理由
* 未確定点
* 次に確認すべきこと

旅行系なら

* 候補の店
* 候補の景色スポット
* ホテルからのアクセス感
* 子連れ向きか
* 車移動しやすいか

まで落とすようにする

9. 状態管理

runStatus

* running
* checkpoint
* continued
* completed
* error

conclusionStatus

* generating
* provisional
* final
* error

ルール

* 2ラウンド後は checkpoint
* ボタン選択前に completed にしない
* ここで終える で final
* 議論を続ける で continued → 次ラウンドへ

10. 受け入れ条件

以下を満たしたらOKです。

UI

* 次アクションが最後まで見える
* 結論カード下部が見切れない
* 続ける / 終える ボタンが見える

フロー

* 2ラウンド後に暫定結論 + ボタンが出る
* 続けると追加議論される
* 暫定結論が更新される
* 終えると最終結論になる

表示

* 日本語UIで英語混在なし
* 送信直後に結論へ飛ばない
* Casual / Soft で自然な文体

論理

* 複合質問の要件を途中で落とさない
* 結論が意思決定支援の粒度に近づく
