Qrooma を以下の内容で修正してください。
今回は、議論ロジックの追加改善より前に、基礎品質の崩れを直すことを優先してください。

今回の最優先課題

1. 複合質問を分解できず、最初の回答で質問の一部しか拾えていない
2. 日本語UIなのに途中から英語が混ざる
3. Run 2以降でも結論が最新内容に更新されない
4. Casual / Soft 設定なのに文体が硬すぎる

⸻

1. 複合質問を要件分解してから回答する

現状の問題

たとえば以下のような入力で、

* 仙台
* 秋保のホテル
* 牛タン
* 喜久福のクリーム大福
* ひょうたん揚げ
* おすすめ観光地
* 今ならではの景色
* 車移動

という複数要素があるのに、最初の回答が一部だけしか拾えていない。
これは議論以前に、入力理解が弱いです。

修正方針

議論開始前に、内部で必ず query decomposition を行ってください。

内部で分解すべきもの

* 食べたいもの
* 行きたい場所
* 条件
* 制約
* 優先度
* 季節性
* 移動手段

例

入力:
「家族旅行で仙台のあきほのホテル。牛タン食べたい。きくふくのクリーム大福、瓢箪揚げ、おすすめ観光地教えて。今ならでは景色を見たい。車。」

内部でこう分解する:

* エリア: 仙台 / 秋保周辺
* グルメ候補: 牛タン / 喜久福 / ひょうたん揚げ
* 観光: おすすめ観光地
* 景色: 今の季節に合う景色
* 条件: 家族旅行 / 車移動

出力ルール

最初の提案では、最低でもユーザー入力の主要要素を全部拾うこと。
全部を深掘りできなくてもよいが、取りこぼさないこと。

期待挙動

初回回答で、

* グルメ
* 観光
* 季節の景色
* 移動条件
    が最低限すべて触れられる状態にする

⸻

2. 日本語UIで英語が混ざる問題を直す

現状の問題

日本語モードなのに途中で

* This fails when ...
* Comparing the options by ...
    のような英語が混ざる

これはモデルの気まぐれではなく、内部テンプレや中間出力がそのまま漏れている可能性が高い。

修正方針

UI言語が日本語のときは、全表示テキストを日本語固定にしてください。

対象

* エージェント本文
* rebuttal
* critique
* round summary
* conclusion
* unresolved / error / continue 系テキスト

実装

* 表示直前に language guard を入れる
* 内部中間データが英語でも、ユーザー向けに出す前に必ず日本語化
* 日本語モードで英語率が一定以上なら再整形する

期待挙動

* 日本語UIでは英語が1文も混ざらない
* とくに Gemini 側で混ざらない

⸻

3. 結論が Run 2 以降で更新されない問題を直す

現状の問題

Run 2 に入って議論内容は更新されているのに、

* 結論カードが前回のまま
* 1回 のまま
* 最新 conclusion が反映されない

原因候補

* conclusion history と latest conclusion の参照先がズレている
* runId と conclusionId が紐づいていない
* summary は更新されるが conclusion 再生成が走っていない
* conclusion 失敗時に前回データを残したままにしている

修正方針

A. データ構造を明確化

最低限、以下を持つ

* runId
* roomId
* conclusionId
* isLatest
* createdAt
* status
* content

B. ルール

* 各 Run ごとに conclusion を必ず生成する
* Run 完了時に、その Run の conclusion を latest に更新する
* 以前の latest は history に残す
* UIは常に current room の latest conclusion を参照する

C. 失敗時

* 新しい conclusion が失敗した場合は、前回 conclusion を latest 扱いにしない
* 状態として error または unresolved を明示する
* 「前回の結論を表示中」など必要なら表示する

期待挙動

* Run 2 後は Run 2 の結論が出る
* Run 3 後は Run 3 の結論が出る
* 履歴も増える
* 結論・1回 の数字も正しく増える

⸻

4. Casual / Soft なのに硬い文体を直す

現状の問題

Casual / Soft 設定なのに、出力がまだ硬い。

例:

* 「〜に依存している」
* 「比較軸は〜であり」
* 「現時点では〜が優先されるべきだろう」
* 「〜の確実性が低い」

これは Casual ではなく、分析レポート調です。

修正方針

文体制御をもっと強く効かせてください。
ただし、中身の判断は弱くしないこと。

Casual + Soft の理想

* 言い方はやわらかい
* でも判断はする
* 断定しすぎない
* 読みやすい
* レポート調ではなく、自然な相談相手っぽい

文体変換ルール

NG寄り

* 「〜に依存している」
* 「現時点では〜が優先される」
* 「〜の確実性が低い」
* 「〜が妥当である」
* 「〜の比較軸において」

目指す表現

* 「この案の弱いところは、〜です」
* 「その条件なら、〜の方が動きやすそうです」
* 「今の情報だと、まだここは絞り切れていません」
* 「いったんはA寄りですが、Bの可能性も残ります」
* 「今回はこっちを先に見るのが自然そうです」

実装ルール

* Debate Logic と Writing Style は分離したまま
* そのうえで Casual / Soft では、後段の表現整形を強く効かせる
* 内容は鋭く、表現だけやわらかくする

期待挙動

Casual + Soft のとき、

* 自然に読める
* 堅すぎない
* でも議論がぬるくならない

⸻

5. 初回回答のフォーマットも改善する

複合質問に対しては、初回回答を少し整理して見せる。

初回回答の形

* まず全体方針を1〜2文
* その後、必要なら項目ごとに短く触れる

例

* 牛タン候補
* スイーツ候補
* 軽食候補
* 景色スポット
* 観光スポット
* 車移動ならの補足

※ ただし表形式ではなく、自然な短文ベースでよい

⸻

6. 受け入れ条件

以下を満たしたらOKです。

入力理解

* 複合質問の主要要素を初回回答で取りこぼさない

言語

* 日本語UIで英語混在が起きない

結論

* Run 2以降も latest conclusion が更新される
* 結論回数も正しく増える
* history と整合する

文体

* Casual / Soft で硬すぎない
* でも判断はちゃんとある
