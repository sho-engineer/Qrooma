Qrooma を以下の方針で改善してください。
今回は 機能追加や大きな仕様変更は行わず、非機能面の安定化 を目的にしてください。

⸻

目的

Qrooma のユーザー向け機能はなるべく変えず、以下を安定させてください。

* Gemini を含む各モデルの出力ブレ
* 英語混在
* モデル名表示のズレ
* 結論更新漏れ
* 再実行時の揺れ
* エラー時の壊れ方
* 出力粒度や長さのばらつき

重要

今回の方針は 機能維持・品質安定化 です。
以下は変えないでください。

* 通常モード / プロンプトモードの基本体験
* 提案 → 検証 → 結論の流れ
* 継続議論 / 最終確定の仕様
* カジュアル / やわらかめ などの設定項目
* 比較軸、残論点、次アクションの表示方針

⸻

基本方針

Qrooma の不安定さは、モデル性能そのものよりも、

* 中間プロンプトの揺れ
* 役割指示の広さ
* 出力後の検品不足
* 状態管理のズレ

に起因している可能性が高いです。

そのため、今回は 前処理・中間入力・後処理・表示整合 を安定化してください。

⸻

1. 中間プロンプトの骨格を固定する

目的

Gemini などに渡す中間入力が毎回少しずつ違うと、出力が大きく揺れます。
自然文のままでよいので、最低限 毎回同じ構成・同じ順序 で渡してください。

必須の順序

以下の順序で中間プロンプトを構成してください。

1. ユーザーの目的
2. 今回決めたいこと
3. 入力から抽出した要求事項
4. 比較軸
5. 現在の有力案 or 現在案
6. 現在の争点
7. 残論点
8. 今回のそのモデルの役割
9. 出力ルール
10. 言語
11. 文体設定

重要

* 項目の有無や並び順を毎回変えない
* summary の粒度を揃える
* 同じ run type では同じテンプレートを使う

⸻

2. 各モデルの役割を広げすぎない

目的

1つのモデルに多すぎる仕事をさせると不安定になります。

対応

今の体験は維持しつつ、内部的には各モデルの責務を狭めてください。

Free（2エージェント）

* GPT: 提案・候補整理
* Gemini: 検証・条件整理

Connect / Pro（3エージェント）

* GPT: 提案
* Claude: 検証
* Gemini: 実行具体化 または 条件整理
    ※ どちらかに寄せる。両方を強く持たせすぎない

禁止

同一モデルに同時に以下を強く求めない

* 比較
* 反論
* 実行化
* 要約
* 文体最適化

⸻

3. ロジックと文体の優先順位を整理する

目的

文体指定が強すぎると、ロジックまで揺れやすくなります。

対応

文体は残してよいですが、内部的には優先順位を明確にしてください。

優先順位

1. 論点に沿って答える
2. 役割を守る
3. 比較軸に触れる
4. 長さ制約を守る
5. 文体設定に合わせる

重要

* Casual / Soft は維持する
* ただしロジックより文体を優先しない
* 文体指定はプロンプト末尾に短く置く

⸻

4. 出力後の検品レイヤーを追加する

目的

モデル出力をそのまま出すのではなく、最低限の品質チェックを通してから表示する

必須チェック

表示前に以下を検品してください。

* UI言語が日本語なのに英語混在していないか
* モデルの役割から外れていないか
* 長すぎないか
* 比較軸が必要な場面で比較に触れているか
* 空欄や途中切れになっていないか
* モデル名ラベルと実体が一致しているか

条件を満たさない場合

* 再整形する
* 再生成する
* 最低限、安全なフォールバック文に置き換える

⸻

5. 日本語固定を強化する

目的

日本語UIで英語混在が起きないようにする

対応

* 表示前の最終レイヤーで language guard を入れる
* 内部で英語テンプレを使っても、ユーザー表示前には必ず日本語化
* Agent A などのラベルも日本語化する
* 英語文率が閾値を超えたら再整形する

期待挙動

* 日本語UIでは全表示が日本語
* critique / rebuttal / conclusion でも英語が漏れない

⸻

6. モデル名表示の整合を取る

目的

Gemini の発言に Claude と出るなど、表示と実体のズレをなくす

対応

* message object に provider / model / role を保存
* UIはその message 自体の metadata を参照する
* 他 run や別 step の metadata を参照しない
* fallback 時はラベルも実際の provider に合わせる

期待挙動

* GPT は GPT
* Gemini は Gemini
* Claude は Claude
* 実際に生成したモデル名と表示が一致

⸻

7. 結論更新の整合を強化する

目的

Run 2 以降で結論が最新に更新されない問題を防ぐ

対応

* 各 run ごとに conclusion を生成する
* latest conclusion は常に最新 run のものを指す
* 結論生成失敗時は、古い conclusion を誤って最新表示しない
* provisional / final / error の状態を明確に分ける

必須データ

* roomId
* runId
* conclusionId
* isLatest
* kind: provisional / final
* status
* createdAt

⸻

8. 再実行時の文脈を絞る

目的

再実行で毎回全文を食わせるとノイズが増え、不安定になります

対応

再実行・継続議論時は、必要な情報だけを渡してください。

最低限渡すもの

* 元質問
* 現在の有力案
* 比較軸
* 残論点
* 今回の役割
* 追加条件（あれば）

なるべく渡さないもの

* 全ラウンド全文
* 冗長な自然文summary
* UI表示用の装飾文

⸻

9. 出力の長さと粒度を揃える

目的

長さや粒度のばらつきを減らす

ルール

* 各エージェント本文: 2〜4文
* Round暫定まとめ: 2〜3文
* 結論: 構造固定
    * 採用
    * 棄却
    * 残論点
    * 次アクション

重要

* 長すぎるときは表示前に圧縮
* 短すぎて中身がない場合は再生成

⸻

10. エラーハンドリングを安定化する

目的

raw error や不自然な壊れ方を防ぐ

対応

* ベンダー raw error は通常UIに出さない
* quota / timeout / auth / unknown を分類して短い文言に変換
* 一部のAIが落ちても、可能なら残りで継続
* conclusion が出せない場合でも、UI全体は壊さない

⸻

11. ログを取りやすくする

目的

不安定さの原因を再現しやすくする

追加したいログ

* 各モデルに渡した最終中間プロンプト
* provider / model / role
* runId / roomId
* conclusion generation input
* language guard 発動有無
* 再整形発動有無
* fallback 発動有無

重要

* ユーザー表示には出さない
* 開発用ログとして残す

⸻

12. 受け入れ条件

以下を満たしたらOKです。

機能面

* 既存機能の流れは変わらない
* 設定項目も維持される

安定性

* 同じ入力で出力粒度が極端にぶれにくい
* 日本語UIで英語混在しない
* モデル名表示が一致する
* Run 2 以降も結論が正しく更新される
* 再実行で不自然な脱線が減る
* raw error がそのまま出ない

出力品質

* 長さが揃う
* 役割がぶれにくい
* 比較軸が必要な場面で比較が出る
* カジュアル設定でも中身が崩れない

⸻

13. 返してほしいもの

1. 不安定さの主因の整理
2. 中間プロンプト骨格をどう固定したか
3. 各モデルの役割をどう整理したか
4. 後処理 / 検品レイヤーの内容
5. ログ追加内容
6. 修正したファイル一覧
7. before / after の出力比較
8. 同一入力を複数回実行した際の安定化例
