今回 fix したい仕様

A. 右側のはみ出しを確実に直す

必須

以下すべてで、右端が画面幅に確実に収まるようにしてください。
	•	Rooms 一覧カード
	•	エラーバッジ
	•	ステータス表示
	•	ルームタイトル + 補助情報
	•	Settings 内のフォーム / セレクト
	•	Landing の各カード
	•	モバイル時の全体レイアウト

条件
	•	見切れない
	•	右端で切れて見えない
	•	横スクロールしない
	•	ただ隠すのではなく、内部レイアウトを調整して収める

方向
	•	card 内 padding / gap / text truncate / badge width を見直す
	•	min-width, flex-shrink, overflow-hidden, truncate, max-width の扱いを整理する
	•	“表示したい要素を全部並べて結果的に切れる” をやめる

⸻

B. チャットの主表示は役割名にする

変更したいこと

チャットバブルの表示名は、モデル名そのものではなく、役割名ベース にしてください。

例

Structured Debate のときは、たとえば
	•	推進
	•	批判
	•	実行
	•	結論

のように見せる

もしくは、MVPで役割文言をまだ完全に決めないなら最低でも
	•	Side A
	•	Side B
	•	Side C
	•	Judge / 結論

などでもよいです

重要
	•	モデル名は詳細情報として控えめに扱う
	•	たとえば
	•	サブラベル
	•	Tooltip
	•	小さい補助表示
などなら可
	•	でも主表示は役割側に寄せたいです

目的

Qrooma は「3モデルが並ぶ場」ではなく、
複数の視点で議論して結論に至る場 に見せたいからです。

⸻

C. APIキー未設定なら議論は動かない前提をUIに反映する

固定方針
	•	Qrooma は BYOK 方式
	•	APIキー未設定では議論実行不可
	•	これは全モード共通
	•	少なくとも利用する provider の APIキーが未設定なら、その構成では run できない

UIで必要なこと

1. 分かりやすい説明
Settings や初回導線で、
	•	APIキーが必要
	•	未設定だと議論が動かない
	•	キーは暗号化保存される
を分かりやすく出す

2. 未設定時の状態表示
	•	“API Key 未設定” バッジは残す
	•	ただし警告感が強すぎない、静かな表示にする
	•	でも動かないことは明確に伝える

3. Run 実行前の案内
少なくとも MVP では、
	•	必要な APIキーが揃っていない場合
	•	送信後に失敗させるだけでなく
	•	送信前 or 送信時に分かる形が理想

できれば
	•	無効状態の説明
	•	CTA: Settings へ
があるとよいです

⸻

D. 2 / 3 エージェント切替と重複禁止も引き続き必要

これは前回のまま継続です。

2 / 3 エージェント切替
	•	UI上に明確に表示
	•	default は 3
	•	2 のときは Side A / B のみ表示
	•	3 のときは Side A / B / C

同一モデル重複禁止
	•	同じ provider + model は重複不可
	•	他サイドで使われている組み合わせは disabled
	•	少なくとも保存時エラーでも可
	•	できれば選択時点で防ぐ

⸻

	•	surface layering を作る
	•	軽い motion を入れる
	•	AIっぽい派手さは減らす
	•	でも無機質・単調にはしない
